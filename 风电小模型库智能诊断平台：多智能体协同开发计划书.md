\# 风电小模型库智能诊断平台：多智能体协同开发计划书



\## 一、项目目标



本项目目标是基于已经整理完成并验证通过的 `littlemodel` 小模型库，开发一个完整的风电小模型库与智能诊断平台。



已知小模型库路径：



```text

C:\\Users\\luzian\\Desktop\\littlemodel

```



平台项目路径：



```text

C:\\Users\\luzian\\Desktop\\wind\_platform

```



`littlemodel` 中已经包含三类模型：



```text

fault\_diagnosis      风电故障诊断模型

rul\_prediction       剩余寿命预测模型

anomaly\_detection    SCADA 异常检测模型

```



每个模型都已经整理为统一结构，并且通过 `model\_registry.json` 暴露统一入口：



```python

predict(input\_path: str, output\_dir: str, options: dict | None = None) -> dict

```



平台开发目标：



```text

1\. 后端能读取 littlemodel/model\_registry.json；

2\. 后端能上传文件并调用三类小模型；

3\. 后端能保存诊断记录和 result.json；

4\. 前端能展示模型库、上传数据、执行诊断、展示结果；

5\. 平台能生成诊断报告；

6\. 后期能接入智能体问答；

7\. 全流程必须经过真实环境测试；

8\. 每个阶段验证通过后，才允许进入下一阶段。

```

\## 二、多智能体角色设计



本项目采用三智能体协同开发模式：



```text

Planner Agent：规划智能体

Dev Agent：执行智能体

Verifier Agent：验证智能体

```



三者职责如下：



| 智能体            | 中文名称  | 核心职责                                 | 是否允许写代码   |

| -------------- | ----- | ------------------------------------ | --------- |

| Planner Agent  | 规划智能体 | 拆分任务、制定阶段目标、维护计划、决定下一阶段是否开始          | 只允许写文档和计划 |

| Dev Agent      | 执行智能体 | 根据 Planner 的阶段任务开发代码、修改代码、自测         | 允许读写代码    |

| Verifier Agent | 验证智能体 | 在真实环境中运行测试、接口验证、页面验证、生成 PASS/FAIL 报告 | 不允许修改业务代码 |



\### 2.1 Planner Agent 职责



Planner Agent 是项目总编排者，但不直接开发核心代码。它负责：



```text

1\. 创建和维护 docs/dev-plan.md；

2\. 创建和维护 docs/main-log.md；

3\. 创建和维护 docs/lessons-learned.md；

4\. 将项目拆分为多个阶段；

5\. 为每个阶段写清楚目标、输入、输出、验收标准；

6\. 调用 Dev Agent 执行当前阶段；

7\. 调用 Verifier Agent 检查当前阶段；

8\. 根据验证结果决定：

&#x20;  - PASS：进入下一阶段；

&#x20;  - FAIL：把测试报告交给 Dev Agent 修复；

9\. 所有阶段结束后，要求 Verifier Agent 做全量测试；

10\. 输出最终项目总结。

```



Planner Agent 不允许：



```text

1\. 不直接修改后端业务代码；

2\. 不直接修改前端业务代码；

3\. 不跳过验证阶段；

4\. 不在 Verifier FAIL 时进入下一阶段；

5\. 不删除 littlemodel 中的模型文件。

```



\### 2.2 Dev Agent 职责



Dev Agent 是开发执行者。它负责：



```text

1\. 阅读当前阶段任务；

2\. 阅读 docs/dev-spec.md、docs/api-spec.md、docs/ui-spec.md；

3\. 阅读 docs/lessons-learned.md；

4\. 编写或修改代码；

5\. 自行运行必要的最小自测；

6\. 输出本阶段修改了哪些文件；

7\. 输出本阶段完成摘要；

8\. 在 Verifier FAIL 后，根据测试报告修复。

```



Dev Agent 必须遵守：



```text

1\. 不改 littlemodel 模型算法；

2\. 不重新训练模型；

3\. 不修改 littlemodel 测试数据；

4\. 只通过 model\_registry.json 和 inference.py:predict 调用小模型；

5\. 不能绕过统一模型调用层；

6\. 每次修改后必须说明修改文件路径；

7\. 如果改动接口，必须同步更新 API 文档。

```



\### 2.3 Verifier Agent 职责



Verifier Agent 是质量检验者。它负责在真实环境中验证平台是否可用。



Verifier Agent 必须只读，不得修改业务代码。允许创建测试报告、截图、测试日志，但不得修改后端、前端和模型库代码。



Verifier Agent 负责：



```text

1\. 运行后端单元测试；

2\. 运行模型库调用测试；

3\. 启动 FastAPI 后端；

4\. 检查 /docs 或 OpenAPI 是否可访问；

5\. 测试 /api/models、/api/upload、/api/diagnose 等接口；

6\. 启动前端；

7\. 用浏览器或 Playwright 测试页面；

8\. 检查上传、诊断、展示、报告生成流程；

9\. 检查错误处理；

10\. 输出 PASS/FAIL 测试报告。

```



Verifier Agent 的报告必须使用统一格式：



```markdown

\# 阶段 N 验证报告



\## 判定

PASS / FAIL



\## 测试环境

\- 操作系统：

\- Python 版本：

\- Node 版本：

\- 后端启动命令：

\- 前端启动命令：



\## 测试项目

| 检查项 | 结果 | 说明 |

|---|---|---|

| ... | PASS/FAIL | ... |



\## 严重问题

1\. ...



\## 中等问题

1\. ...



\## 轻微问题

1\. ...



\## 是否允许进入下一阶段

允许 / 不允许

```



判定规则：



```text

PASS：

\- 当前阶段所有核心功能通过；

\- 无严重问题；

\- 中等问题不超过 1 个；

\- 不影响进入下一阶段。



FAIL：

\- 有任何严重问题；

\- 有 2 个及以上中等问题；

\- 当前阶段核心功能无法真实运行；

\- 接口跑不通；

\- 前端无法启动；

\- 模型调用失败；

\- 报告无法生成。

```

\## 三、全局开发原则



\### 3.1 阶段门禁原则



每个阶段必须遵循：



```text

Planner 制定阶段任务

→ Dev 执行开发

→ Dev 自测

→ Verifier 真实环境验证

→ Verifier 输出 PASS/FAIL

→ PASS 才能进入下一阶段

→ FAIL 必须回到 Dev 修复

→ 修复后 Verifier 重新测试

```



不允许跳过 Verifier。



\### 3.2 修正循环原则



每个阶段最多允许 5 轮修正：



```text

第 1 轮开发 → 验证

FAIL → 第 1 轮修复 → 复测

FAIL → 第 2 轮修复 → 复测

...

最多第 5 轮

```



第 5 轮仍 FAIL 时：



```text

1\. 不允许强制进入下一阶段；

2\. Planner 必须暂停项目；

3\. 在 docs/blockers.md 中记录阻塞原因；

4\. 输出需要人工介入的问题清单。

```



本项目是平台工程，不允许低质量强制通过。



\### 3.3 文件即记忆原则



所有阶段产出必须写入文件，不依赖对话上下文。



必须维护以下文件：



```text

C:\\Users\\luzian\\Desktop\\wind\_platform\\docs\\dev-plan.md

C:\\Users\\luzian\\Desktop\\wind\_platform\\docs\\main-log.md

C:\\Users\\luzian\\Desktop\\wind\_platform\\docs\\lessons-learned.md

C:\\Users\\luzian\\Desktop\\wind\_platform\\docs\\api-spec.md

C:\\Users\\luzian\\Desktop\\wind\_platform\\docs\\ui-spec.md

C:\\Users\\luzian\\Desktop\\wind\_platform\\docs\\test-plan.md

C:\\Users\\luzian\\Desktop\\wind\_platform\\docs\\blockers.md

```



\### 3.4 日志原则



Planner 必须在 `docs/main-log.md` 中记录每个关键动作：



```markdown

\- 260603 2100 项目启动

\- 260603 2105 Phase 0 开始：项目初始化

\- 260603 2120 Phase 0 开发完成，启动验证

\- 260603 2130 Phase 0 验证 PASS，进入 Phase 1

\- 260603 2200 Phase 1 验证 FAIL，返回 Dev 修复

```



时间格式使用：



```text

```



\### 3.5 经验库原则



每次 Verifier 发现问题，Dev 修复后必须更新：



```text

```



记录格式：



```markdown

\## 前端展示

```
