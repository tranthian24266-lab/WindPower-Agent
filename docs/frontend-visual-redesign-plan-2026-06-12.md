# 风电智能体前端视觉改造方案

更新时间：2026-06-12  
适用仓库：`C:\Users\luzian\Desktop\windpower_agent3`  
适用范围：`frontend/` 下 React + TypeScript + Vite + Tailwind v4 单页应用

## 1. 文档目标

本文档不是泛泛而谈的“界面优化建议”，而是一份可直接执行的前端视觉改造方案，目标是把当前前端从“整洁可用的业务后台”提升为“有明显品牌识别度、具有工业智能中枢气质、适合路演和长期使用的风电智能体平台”。

本文档重点回答以下问题：

1. 当前前端主要问题在哪里。
2. 整体视觉方向应改成什么样。
3. 全局样式、页面布局、组件体系分别要改哪些地方。
4. 每个核心页面应如何重构。
5. 修改应落到哪些文件。
6. 如何分阶段实施、如何验证是否改到位。

## 2. 当前前端现状判断

### 2.1 当前优点

当前前端已经具备较好的工程基础：

- 路由结构清晰，页面覆盖面完整。
- 全局已经有统一的卡片、按钮、状态、表格、筛选布局。
- 信息组织整体有秩序，不杂乱。
- 颜色方案已经有一定风电行业感，主色偏绿色，辅色偏橙色。
- 核心页面已经具备真实业务数据，不是纯静态壳子。

### 2.2 当前主要问题

当前问题不在“是否好用”，而在“视觉表达不够强、不够分层、不够像成熟产品”。

核心问题如下：

1. 全站过于平均。
   - 几乎所有页面都在重复同样的 `PageHeader + content-panel + list-card + metric-card`。
   - 页面之间缺少视觉角色差异，首页、模型库、知识库、问答页看起来过于相似。

2. 品牌识别弱。
   - 左侧栏的 `WP` 品牌标识过于临时。
   - 缺少“风电 + 智能体 + 工业诊断”的统一品牌语言。
   - 缺少可以让人一眼记住的主视觉区域。

3. 首页没有“指挥台”气质。
   - 首页当前是合理的信息拼装，但不是强首页。
   - 缺少一个能代表平台能力的英雄区。
   - 四个指标卡的信息价值存在，但视觉冲击力不够。

4. 页面层级拉不开。
   - 标题、说明、卡片、列表、筛选区都比较平。
   - 重要信息没有被足够突出，次要信息也没有很好退后。
   - 同一页面内部缺少“主区、次区、控制区、证据区”的层级。

5. 高信息密度页面仍像“内部工具”。
   - `/models` 功能很多，但不够高级，像管理后台，而不是模型资产中台。
   - `/chat` 长 case id 直接铺开，视觉噪音很大。
   - `/knowledge` 空状态较弱，状态面板缺少仪表盘感。

6. 视觉语言还不够独特。
   - 当前圆角、渐变、背景、按钮都偏温和安全。
   - 工业感、实时感、控制中枢感还不够。
   - 没有建立“智能体正在工作”的动态感知。

## 3. 目标视觉方向

## 3.1 总体方向

建议把整站改造成以下风格：

> 风电智能体控制中枢  
> 既有工业监控平台的秩序感，又有智能系统的实时反馈感，同时保留适合中文业务平台的清晰可读性。

这不是赛博朋克，也不是传统深色运维台，而是：

- 深色、稳定、专业的全局骨架。
- 明亮但克制的内容区。
- 强主视觉的首页与关键模块头部。
- 明确区分“数据状态”“执行动作”“证据来源”“风险等级”。

## 3.2 建议视觉关键词

- 工业中控
- 风场流线
- 智能调度
- 风险态势
- 证据驱动
- 冷静、克制、可信

## 3.3 建议色彩方向

保留现有绿色基因，但整体升级为“深海蓝绿 + 风机青绿 + 安全橙 + 警戒红”。

建议主色体系：

- 页面深背景主色：`#0f1b1f`
- 侧栏深背景：`#13242b`
- 内容主背景：`#eef4f3`
- 卡片白底：`rgba(255,255,255,0.88)`
- 主强调色：`#1ea672`
- 辅强调色：`#3bb3c3`
- 点睛色：`#ff9a4d`
- 危险色：`#d85a4a`
- 文字主色：`#14252a`
- 次级文字：`#61757b`

注意：

- 不要走紫色科技风。
- 不要纯黑高对比。
- 不要做成泛 SaaS 模板。

## 3.4 建议字体与排版方向

当前字体栈可读，但个性不足。建议：

- 中文优先继续使用 `"Microsoft YaHei UI"`, `"PingFang SC"`, `"Noto Sans SC"`。
- 英文和数字可增加更有工业感的备选，例如 `"Bahnschrift"`, `"Segoe UI Variable"`，用于大号指标和品牌字。
- 首页大标题和核心指标采用更紧凑的字距。
- 页面标题与 section 标题的层级差进一步拉开。

排版原则：

- 大标题更大、更短、更像产品，而不是说明文。
- 次级说明更短，不堆句子。
- 指标数字更像“仪表读数”，而不是普通文本。

## 4. 总体改造策略

建议遵循以下总原则：

1. 不重写业务逻辑，优先做视觉和信息架构升级。
2. 先全局，后页面。
3. 先改骨架和 token，再改核心页面，再扫尾次要页面。
4. 保持所有现有路由、核心交互、数据接口兼容。
5. 优先提升首页、诊断页、模型库、问答页、知识库。

## 5. 全局设计系统改造

## 5.1 目标

先把全局视觉地基搭好，否则每个页面单独改会风格漂移。

## 5.2 主要修改文件

- `frontend/src/styles.css`
- `frontend/src/components/Layout.tsx`
- `frontend/src/components/PageHeader.tsx`
- `frontend/src/components/MetricGrid.tsx`
- 如需要新增：
  - `frontend/src/components/SectionCard.tsx`
  - `frontend/src/components/StatPanel.tsx`
  - `frontend/src/components/EmptyState.tsx`
  - `frontend/src/components/PageHero.tsx`

## 5.3 全局样式层面要改什么

### 5.3.1 背景系统

当前背景偏浅，建议改成“双层背景”：

- 应用最外层背景更深，形成“中控台外壳”。
- 内容区保留明亮半透明卡片，形成对比。
- 页面背景增加更细腻的流线型径向渐变，而不是简单的两块 radial gradient。

目标效果：

- 一眼看上去更像产品，而不是普通后台。
- 深侧栏和浅内容区形成稳定对比。
- 既有工业感，又不牺牲阅读舒适性。

### 5.3.2 圆角体系

当前圆角较多且偏统一，建议分层：

- 大容器：`28px`
- 中卡片：`22px`
- 小控件：`14px`
- pill/badge：`999px`

目标效果：

- 页面更有秩序。
- 不同层级一眼可分。

### 5.3.3 阴影体系

当前阴影较轻，建议建立两套：

- 普通卡片阴影：轻、柔、低对比。
- 焦点卡片阴影：更深、更集中，配合 hover 和 selected 状态。

不要做：

- 黑色重阴影。
- 所有卡片都浮起来。

### 5.3.4 边框体系

当前边框偏普通。建议：

- 卡片主边框改为低透明度冷色边框。
- 激活态使用青绿或橙色细描边。
- 特殊卡片允许顶部或左侧出现 2px 强调条。

### 5.3.5 动效体系

应增加少量有意义的动效：

- 页面首次进入时，hero 和第一屏卡片做轻微上浮显现。
- hover 卡片增加轻微位移和阴影变化。
- 按钮点击保持克制，不做夸张弹跳。
- 智能体运行中、加载中、引用依据展开时，可使用线性流动感动画。

目标效果：

- 用户感知到“系统在工作”。
- 不做花哨装饰。

## 5.4 布局层面要改什么

### 5.4.1 侧边栏

当前问题：

- 品牌区过弱。
- 导航区功能齐全，但没有节奏。
- 页面切换的高级感不够。

建议改法：

1. 品牌区重做。
   - `WP` 不再只是字母块。
   - 变成更像产品徽章的标识，例如风机叶片抽象图形 + `WindPower Agent`.
   - 品牌区下加一句更短副标题，如“风电智能诊断中枢”。

2. 导航项升级。
   - 每个导航项加简洁图标。
   - 当前激活项使用更明显的高亮底、发光边、左侧引导线。
   - 未激活项透明度更低。

3. 侧栏底部卡片升级。
   - `ApiKeySessionCard` 不应像临时调试块。
   - 应做成“会话控制卡”风格，标题更明确，说明更短，输入框更整洁。

4. 侧栏增加顶部与底部视觉锚点。
   - 顶部代表品牌和导航。
   - 底部代表当前会话环境。

目标效果：

- 左栏看起来像一个真正产品壳层，而不是导航容器。

### 5.4.2 主内容容器

建议：

- 主内容区最大宽度按页面类型区分，不要一把梭。
- 首页、知识库、模型库允许更宽。
- 聊天页和诊断页可适度收束，提升聚焦感。

可以考虑：

- 首页和模型库：`1240px`
- 诊断页和聊天页：`1120px`
- 报告阅读类页面：更宽但边距更大

## 5.5 通用组件层面要改什么

### 5.5.1 `PageHeader`

当前问题：

- 所有页面头部都很像。
- 头部没有根据页面角色变化。

建议：

- 把 `PageHeader` 升级为可变体组件。
- 至少支持：
  - `overview`
  - `workspace`
  - `operations`
  - `detail`

每种变体变化：

- 背景
- 标题尺寸
- 是否显示辅助状态区
- action 区布局

### 5.5.2 `MetricGrid`

当前问题：

- 指标卡过于标准化，缺少权重差异。

建议：

- 支持大卡、小卡、趋势卡三种形态。
- 可选图标、变化说明、状态标签。
- 首页第一屏至少有 1 个大卡，其余为次级卡。

### 5.5.3 列表卡片

建议把当前 `list-card` 细分：

- 信息列表卡
- 可点击导航卡
- 风险卡
- 证据卡
- 版本卡

因为这些卡片在视觉职责上已经不同，不应继续共用几乎一样的样式。

### 5.5.4 状态组件

建议统一状态展示语言：

- 正常：青绿
- 关注：琥珀
- 高风险：橙红
- 失败：深红
- 运行中：青蓝

涉及组件：

- `RiskBadge`
- `StatusBanner`
- 加载按钮
- step/routing/validation 状态标签

## 6. 首页改造方案

对应文件：

- `frontend/src/pages/DashboardPage.tsx`
- 可能新增首页专属局部组件

## 6.1 目标

把首页从“统计概览页”升级为“风电智能体总览控制台”。

## 6.2 主要问题

- 头部过于普通。
- 4 个指标卡没有主次。
- 模型概览和最近案例虽然有信息，但缺少形象表达。
- 首页没有“这就是平台核心能力”的主视觉。

## 6.3 建议结构

首页建议改成 5 个区块：

1. 指挥台 Hero
2. 关键态势指标
3. 智能体能力面板
4. 最近案例与风险动态
5. 快速入口区

## 6.4 Hero 区怎么改

左侧内容：

- 小标题：平台总览
- 大标题：建议从“统一监控风电诊断能力”改成更有产品感的表达
  - 备选 1：`风电智能体诊断中枢`
  - 备选 2：`统一调度诊断、问答与模型能力`
  - 备选 3：`让风电诊断从结果页升级到决策流`
- 副标题一句话，控制在 24 到 36 字。

右侧内容：

- 不用纯文本。
- 建议做一个数据化视觉模块，例如：
  - 风险环形分布
  - 模型覆盖图
  - 智能体运行流线图
  - “上传 -> 诊断 -> 报告 -> 问答”工作流示意

Hero 行动区：

- 主按钮：开始诊断
- 次按钮：打开模型库
- 可增加小状态条：在线模型数、最近案例数、系统状态

## 6.5 指标区怎么改

当前 4 个指标保留，但结构调整：

- 左上大卡：历史案例数或今日处理量
- 其余 3 卡：在线模型、故障模型、异常模型

如可以从接口拿更多数据，建议增加：

- 高风险案例数
- 最近 24 小时处理数
- 正在运行的 agent 任务数

如果暂时拿不到，不新增假数据。

## 6.6 模型概览区怎么改

现在只是列表。建议：

- 每个模型卡增加小图标或任务类型色块。
- 模型名、用途说明、任务类型分三层展示。
- 加一个“平台能力覆盖”副标题。

目标效果：

- 用户一眼知道平台不只是一个模型，而是一个能力矩阵。

## 6.7 最近案例区怎么改

建议：

- 风险 badge 更醒目。
- 文件名和时间层级更明确。
- 每条案例左侧可以加小型状态轨迹线或风险色标。
- 列表顶部加一句说明，如“最新诊断动态”。

## 6.8 快速入口区怎么改

新增一个 3 到 4 项的快捷入口区：

- 新建诊断
- 打开问答
- 查看知识库
- 查看评测或审核

让首页更像“进入各工作台的入口”。

## 7. 诊断页改造方案

对应文件：

- `frontend/src/pages/DiagnosisPage.tsx`

## 7.1 目标

把当前诊断页从“表单 + 上传”改为“步骤式任务工作台”。

## 7.2 当前问题

- 页面合理，但像普通上传页。
- “选择任务”和“上传文件”之间关系不够强。
- 主按钮视觉权重还不够。
- 高级选项展开后仍像调试配置，不像专业控制项。

## 7.3 目标结构

建议改成清晰的三段式：

1. 任务选择
2. 样本上传
3. 执行动作与运行状态

## 7.4 任务选择区怎么改

任务卡保留 3 个，但升级为：

- 带图标
- 带场景说明
- 带输入类型提示
- 带风险/用途说明

例如每个任务卡内部固定四层：

- 任务名
- 一句用途说明
- 支持文件类型
- 典型输出

选中态要更强：

- 边框高亮
- 顶部色带
- 轻微阴影提升

## 7.5 上传区怎么改

当前上传区可以明显放大。

建议：

- 改成更像拖拽上传台的区域。
- 增加图标、文件格式提示、当前已选文件信息。
- 文件选中后展示“文件名 + 文件类型 + 可执行任务提示”。

## 7.6 高级选项怎么改

当前高级选项更像调试面板。建议：

- 视觉上弱化为“专家控制项”。
- 默认折叠。
- 展开区域使用与主面板略不同的背景。
- 说明文案明确这是“模型路由偏好”，不是普通用户必须配置项。

## 7.7 运行状态区怎么改

当前只有状态文本。建议增强为：

- 主按钮右侧显示运行状态。
- 运行时展示一个轻量级进度感组件。
- 状态文案分阶段：
  - 已选择任务
  - 已选择样本
  - 正在上传
  - 正在执行诊断
  - 正在跳转案例

## 8. 模型库页面改造方案

对应文件：

- `frontend/src/pages/ModelLibraryPage.tsx`

## 8.1 目标

把模型库从“后台管理页”提升为“模型资产中台”。

## 8.2 当前问题

- 功能很全，但视觉节奏过于密集。
- 顶部筛选区略普通。
- 左侧列表与右侧详情已经有雏形，但高级感还不够。
- “版本”“验证”“路由”是强能力模块，现在视觉区分还不够。

## 8.3 顶部头部怎么改

头部建议强化：

- 标题可改为更强表达，如“平台模型资产中心”。
- action 区按钮分主次：
  - 主按钮：同步目录
  - 次按钮：回到首页

头部下可以增加一句短说明：

- “统一管理模型版本、验证状态、别名路由与上线资产”

## 8.4 指标区怎么改

四个指标卡不应等权。

建议：

- 目录家族数作为主指标
- 生产中版本 / 待验证模型作为次指标
- 当前页容量降权或挪到次级说明

可以引入小趋势说明：

- 生产稳定
- 无待验证阻塞
- 当前列表筛选结果

## 8.5 筛选区怎么改

建议：

- 搜索框更长、更主导。
- 筛选器做成同一横向控制条。
- 当屏幕较宽时呈单行高密度控制。
- 增加“已启用筛选条件”可视提示。

目标效果：

- 像真正的资产筛选台，而不是一组普通表单。

## 8.6 左侧模型列表怎么改

每张模型卡增加层级：

- 顶部：模型名 + 任务类型
- 次级：family code
- 中部：一句摘要
- 底部：版本状态、验证状态、别名、最近验证时间

建议增强：

- 当前选中卡片用明显边框 + 局部渐变背景。
- 状态标签更精致。
- 不同任务类型可使用不同细微主题色。

## 8.7 右侧详情怎么改

右侧应更像一个“资产详情控制面板”。

建议分成三层：

1. 头部摘要
2. tab 切换
3. tab 内容卡片

### 8.7.1 头部摘要

建议加入：

## 10.1 目标

## 15. 视觉细节硬约束

执行时应遵守以下约束：

1. 不要使用通用紫色科技风。
2. 不要把页面做成漆黑的运维后台。
3. 不要引入花哨、无意义的动效。
4. 不要只改颜色，不改层级和布局。
5. 不要用过多渐变导致廉价感。
6. 不要把所有页面都做成完全一样的卡片堆。
7. 必须兼顾桌面端和移动端。
8. 优先保留已有业务接口和现有页面职责。

## 16. 验收标准

本轮前端改造完成后，至少应满足以下标准：

1. 首页一眼看上去就是平台首页，而不是普通概览页。
2. 左侧导航有稳定品牌感，而不是临时后台样式。
3. 诊断页具备流程工作台感。
4. 模型库具备模型资产中心感。
5. 知识库具备索引控制台感。
6. 问答页具备案例协同分析感。
7. 页面之间视觉统一，但角色不同。
8. 在 1280 宽度和移动端宽度下都可用。
9. `npm run build` 必须通过。
